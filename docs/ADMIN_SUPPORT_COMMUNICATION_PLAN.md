# Связь администратора с клиентом и экспертом

## Цель

Сделать поддержку единой и понятной: пользователь создает обращение, администратор видит его в одном рабочем месте, отвечает там же, а клиент или эксперт получает ответ в своем центре поддержки. Отдельные "чаты поддержки" не должны жить параллельно с обращениями, потому что это создает дубли, потерю истории и непрочитанные сообщения не в том месте.

## Текущее состояние

- Пользовательский центр поддержки работает через `SupportRequest`, `Claim` и арбитражные дела.
- Админская секция "чаты поддержки" использует отдельный endpoint `/admin-panel/support-chats/`, который возвращает обычные `Chat` с условным пользователем поддержки.
- Ответ администратора в `/admin-panel/support-chats/<id>/send-message/` не попадает в `SupportRequest.messages`, поэтому история обращения и история "чата" расходятся.
- У `SupportRequest` уже есть сообщения, статусы, приоритет, назначенный администратор и активность, поэтому именно он должен быть основной сущностью поддержки.
- Файлы, прочитанность и realtime для обращений поддержки пока не унифицированы.

## Целевая модель

Основная сущность: `SupportRequest`.

Поток:

1. Клиент или эксперт создает обращение из центра поддержки, жалобы, обжалования отзыва или системного сценария.
2. Администратор видит обращение в едином inbox.
3. Администратор берет обращение в работу, отвечает, меняет статус, назначает ответственного или передает в арбитраж.
4. Пользователь видит ответ в том же обращении и получает уведомление с action-кнопкой "Открыть обращение".
5. Все сообщения, файлы и действия сохраняются в одной ленте обращения.

## Этап 1. Единый inbox обращений в админке

Цель: убрать разрыв между "чатами поддержки" и обращениями хотя бы в интерфейсе администратора.

Что делаем:

- Админская секция поддержки читает `/admin-panel/support-requests/`, а не `/admin-panel/support-chats/`.
- Отправка ответа из админского интерфейса идет в `/admin-panel/support-requests/<id>/send_message/`.
- Сообщения отображаются из `SupportRequest.messages`.
- Список показывает тему, пользователя, статус, приоритет, последнее сообщение и время обновления.
- Endpoint `SupportRequestSerializer` отдает удобные поля для inbox: последнее сообщение, время последнего сообщения, количество сообщений.

Критерий готовности:

- Ответ администратора появляется в истории обращения пользователя.
- В админке нет второго независимого потока сообщений для поддержки.
- Старые endpoint-ы `support-chats` остаются для совместимости, но новый UI их не использует.

## Этап 2. Прочитанность и счетчики

Цель: убрать ложные непрочитанные обращения и сделать счетчики доверяемыми.

Статус: частично реализовано для `SupportRequest`. Для `Claim` и арбитражных дел прочитанность нужно добавить отдельным подэтапом, потому что там другие модели сообщений.

Что делаем:

- Добавить в `SupportMessage` поля прочитанности для стороны пользователя и стороны админки.
- Добавить endpoint `mark_read` для обращения.
- Считать `unread_count` отдельно для администратора и пользователя.
- Помечать обращение прочитанным при открытии карточки.
- Не создавать unread для закрытых/недоступных веток.

Реализовано:

- В `admin_panel.SupportMessage` добавлены `read_by_user` и `read_by_admin`.
- Ответ администратора становится непрочитанным для пользователя.
- Сообщение пользователя становится непрочитанным для администратора.
- `/admin-panel/support-requests/<id>/mark_read/` помечает сообщения прочитанными для текущей стороны.
- Админский inbox вызывает `mark_read` при открытии обращения.
- Пользовательский центр обращений вызывает `mark_read` при открытии `SupportRequest`.
- В пользовательском списке обращений появился тег `Новых: N`.

Критерий готовности:

- Если администратор или пользователь открыл обращение, счетчик непрочитанных пропадает.
- Системные уведомления не создают ложный unread в местах, где их нельзя прочитать.

## Этап 3. Файлы и скриншоты в обращениях

Цель: поддержка должна принимать доказательства, скриншоты и вложения без обходных путей.

Что делаем:

- Добавить вложения к `SupportMessage`.
- Поддержать multipart-отправку сообщений.
- Отображать файлы в пользовательском центре поддержки и в админке.
- Добавить ограничения размера и типа файла.

Критерий готовности:

- Пользователь и администратор могут отправить текст с файлами.
- Файлы видны в общей истории обращения.

## Этап 4. Уведомления и action-кнопки

Цель: каждое уведомление должно вести к следующему действию.

Что делаем:

- Для ответа администратора отправлять уведомление "Открыть обращение".
- Для нового обращения отправлять администратору уведомление "Открыть в админке".
- Для претензий, арбитража и жалоб использовать те же action-поля.
- Дедуплицировать одинаковые уведомления в коротком окне времени.

Критерий готовности:

- Пользователь или администратор переходит из уведомления сразу в нужное обращение.
- Нет дублей одинаковых уведомлений по одному событию.

## Этап 5. Realtime и аудит

Цель: поддержка должна ощущаться живой и управляемой.

Что делаем:

- Подключить websocket-события для новых сообщений и смены статуса.
- Добавить ленту активности: создано, взято в работу, ответ, назначение, завершение, передача в арбитраж.
- Добавить SLA-поля: время первого ответа, время решения, просрочено/не просрочено.

Критерий готовности:

- Новые ответы появляются без перезагрузки.
- Руководитель видит нагрузку и скорость реакции поддержки.

## Этап 6. Роли администратора и эксперта

Цель: роли должны видеть только нужные обращения и действия.

Что делаем:

- Уточнить доступы для `admin`, `director`, ответственных сотрудников и владельца обращения.
- Для эксперта показать обращения, связанные с его заказами, отзывами и блокировками.
- Для клиента показать обращения по заказам, оплатам, качеству и арбитражу.
- В админке добавить фильтры по роли заявителя: клиент, эксперт, партнер.

Критерий готовности:

- Эксперт и клиент видят только свои обращения.
- Администратор видит всю поддержку.
- Директор видит аналитику и может подключаться к сложным обращениям.
