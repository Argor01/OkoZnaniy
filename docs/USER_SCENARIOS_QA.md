# QA: пользовательские сценарии OkoZnaniy

Дата составления: 2026-07-23

Цель документа: пройти ключевые сценарии платформы от лица всех ролей и зафиксировать баги, несостыковки frontend/backend, неактуальные API, слабую обработку ошибок, визуальные и адаптивные проблемы.

## Тестовое окружение

- Frontend: `http://localhost:5173`
- Backend API: `http://localhost:8000/api`
- Docker services: `backend`, `frontend`, `postgres`, `redis`, `celery`
- Тестовые аккаунты:
- Клиент: `client@test.com` / `test123`
- Эксперт: `expert@test.com` / `test123`
- Администратор: `administrator@test.com` / `test123`
- Директор: `director@test.com` / `test123`
- Арбитр: `arbitrator@test.com` / `test123`
- Партнер: `partner@test.com` / `test123`

## Условные обозначения

- `PASS`: сценарий пройден без замечаний.
- `FAIL`: сценарий не работает или ломает бизнес-логику.
- `WARN`: работает, но есть риск, UX-дефект или неполная обработка ошибки.
- `TODO`: еще не проверено руками/автотестом.

## Общие критерии проверки

- Страница не должна падать в ErrorBoundary.
- Frontend не должен дергать несуществующий endpoint.
- Ошибки API должны показываться пользователю понятным русским текстом.
- После успешного действия UI должен обновляться без ручной перезагрузки.
- Темная тема должна сохранять контраст текста, карточек, toast/message, modal, dropdown, select, tooltip.
- На ширинах `360`, `390`, `768`, `1024`, desktop элементы не должны накладываться, выходить за viewport или требовать горизонтальный скролл, кроме таблиц/таймлайнов, где он явно предусмотрен.
- WebSocket ошибки не должны блокировать базовую работу через REST.
- Непрочитанные сообщения должны считаться только по реально читаемым сообщениям.
