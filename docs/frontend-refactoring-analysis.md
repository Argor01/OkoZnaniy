# Анализ фронтенда OkoZnaniy — рекомендации по рефакторингу

## Общая статистика
- **136 602 строки** кода (TSX + TS + CSS)
- **227 TSX**, **142 TS**, **260 CSS** файлов
- **19 feature-модулей**
- **49 файлов > 500 строк** (требуют декомпозиции)

## Что сделано хорошо
- Feature-based архитектура (`features/admin`, `features/director`, `features/expert` и т.д.)
- CSS Modules для компонентной стилизации (198 файлов)
- Shared UI-компоненты (`AppButton`, `AppCard`, `AppInput`, `AppSelect`, `AppSpinner`, и др.)
- React Query для управления серверным состоянием
- Отдельный API-слой для каждого feature

### 1. Гигантские компоненты
| Файл | Строки | Рекомендация |
|------|--------|-------------|
| `MessageModalNew.tsx` | **3 903** | Разбить на 10+ подкомпонентов (шапка, тело, формы, файлы, статусы) |
| `OrderDetail.tsx` | 1 299 | Вынести табы/секции в отдельные компоненты |

| + ещё 43 файла > 500 строк | | |

**Порог:** компонент > 300 строк — кандидат на декомпозицию.

### 2. Дублирование чат-компонентов
Три почти идентичных компонента чата:
- `DirectorChatsSection.tsx` — 564 строки
- `PartnerChatsSection.tsx` — 593 строки
- `SupportChatsSection.tsx` — 540 строк
- `AdminChatsSection.tsx` — 285 строк

**Рекомендация:** Создать общий `ChatSection` компонент с пропсами для настройки (API endpoint, роль пользователя, заголовок). Сэкономит ~1 500 строк кода и упростит поддержку.

### 3. Смешанные подходы к стилям
- **198 CSS Module файлов** (правильный подход)
- **62 глобальных CSS файла** (12 250 строк в `styles/`)
  - `landing.css` — 3 102 строки
  - `admin-dashboard.css` — 2 357 строк
  - `director.css` — 1 013 строк
  - `login.css` — 962 строки

**Рекомендация:** Перевести глобальные стили в CSS Modules при соответствующих компонентах. Это устранит конфликты имён классов и упростит поддержку.

### 4. Дублирование feature-папок
- `features/arbitration/` (api, components, pages)
- `features/arbitrator/` (api, components, pages, types)

Два модуля для связанной функциональности. **Рекомендация:** объединить в один `features/arbitration/`.

## Средние проблемы

### 5. console.log в продакшен-коде
Найдено в **15+ файлах** (MessageModalNew, Login, AdminLogin, client.ts, и др.). Часть обёрнута в `debug_api` проверки, но не все.

**Рекомендация:** Удалить все голые `console.log/warn/error`. Использовать единый логгер с проверкой `import.meta.env.DEV`.

### 6. Использование `any` типа
Найдено в **15+ файлах**: MessageModalNew (7 случаев), ExpertApplication (4), ApplicationModal (3), AdminLogin (3), helpers.ts (4).

**Рекомендация:** Заменить на правильные типы. Включить `noImplicitAny` в tsconfig.

### 7. Нет Error Boundaries
Баг с белым экраном (React error #310) показал, что ошибка в одном компоненте роняет всю страницу. Нет Error Boundaries вокруг секций дашбордов.

**Рекомендация:** Добавить `ErrorBoundary` компонент вокруг каждой секции дашборда, чтобы ошибка в одном блоке не ломала весь интерфейс.

### 8. Монолитный файл маршрутов
`AppRoutes.tsx` — 369 строк, все маршруты в одном файле.

**Рекомендация:** Разделить по feature-модулям: `adminRoutes`, `directorRoutes`, `expertRoutes`, `orderRoutes`, и собирать в главном файле.

## Низкий приоритет

### 9. Нет lazy loading для маршрутов
Все компоненты страниц импортируются синхронно. При текущем размере бандла это увеличивает время начальной загрузки.

**Рекомендация:** `React.lazy()` + `Suspense` для страничных компонентов.

### 10. Нет unit-тестов
Папки `__tests__` и тестовые файлы отсутствуют.
